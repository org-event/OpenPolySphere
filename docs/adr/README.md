# Architecture Decision Records (ADR)

Короткие заметки **почему** принято решение, а не только **что** сделано в коде.
Живут рядом с репозиторием; один ADR — одно долгоживущее решение (~10 минут на запись).

## Когда писать

- Решение затрагивает несколько модулей или ОС.
- Через полгода без контекста выглядит странно или «over-engineered».
- Есть отвергнутые альтернативы с вескими причинами.

Не писать ADR на багфиксы, переименования и одноразовые правки.

## Шаблон

Скопируйте в `NNNN-краткое-имя.md`:

```markdown
# [Название решения]

Дата: YYYY-MM

## Контекст

[Ситуация, ограничения, что ломалось или грозило сломаться]

## Решение

[Что конкретно делаем в коде / процессе]

## Почему не альтернативы

- **Вариант A** — …
- **Вариант B** — …

## Последствия

**Плюсы:** …

**Минусы:** …

## Связанные файлы

- …
```
